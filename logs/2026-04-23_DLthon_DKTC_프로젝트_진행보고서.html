3-3. 기타 발견

  • 중복 104건 발견 (기타 괴롭힘에 집중) → 제거 후 3,846건
  • 길이 Train 평균 227자, Test 평균 214자 → 유사한 분포
  • 토큰 95% 데이터가 128 토큰 이내 → max_length=128 설정 근거

4. 합성 데이터 생성 과정

4-1. 전략 설계

Train에 없는 '일반 대화' 클래스를 위해 Claude를 활용한 합성 데이터 생성 전략 채택:

일반 대화 (10개 주제 x 100건):

일상/날씨, 음식/맛집, 취미/여가, 학교/공부, 직장 일상, 쇼핑/패션, 연애/친구, 건강/운동, 뉴스/시사, 가족/집

Hard Negative (200건):

위협 클래스와 혼동될 수 있지만 실제로는 일반인 대화. 직장 일상(70) + 돈 관련 일상(70) + 가벼운 말다툼(60)

4-2. 길이 문제 발견과 해결

문제 발견: 1차 생성 데이터의 평균 길이가 66~82자로 원본 Train(227자) 대비 1/3 수준. 모델이 "짧으면 일반 대화"로 학습할 위험.

5. 모델 선택과 학습

5-1. 모델 선택 근거

beomi/KcELECTRA-base 선정

  • ELECTRA 아키텍처: 모든 토큰을 학습하여 BERT 대비 높은 샘플 효율성
  • 한국어 댓글 사전학습: 구어체/비격식 텍스트에 특화 (위협 대화 도메인 적합)
  • 모델 크기: 109M params → CPU 학습 가능

5-2. 전처리 파이프라인

def preprocess(text):
    text = text.replace('\n', ' ')           # 핵심: newline 정규화
    text = re.sub(r'\s+', ' ', text).strip()  # 다중 공백 제거
    return text

Train과 Test 모두 동일 전처리 적용. 가장 중요한 단일 처리 단계.

5-3. 학습 인프라 이슈와 해결

이슈원인해결
MPS 학습 극도로 느림 Apple Silicon MPS 백엔드의 ELECTRA 최적화 부족 (스텝당 30~100초) CPU 전환 → 스텝당 1.1초
학습 중 속도 급격 저하 체크포인트 17GB 누적으로 디스크 25GB → 스와핑 발생 체크포인트 삭제 + save_total_limit=2
max_len=256 실패 CPU에서 긴 시퀀스 처리 시 메모리/속도 한계 max_len=128 유지 (95% 커버)

진단: Validation에서는 합성 데이터를 완벽히 분류하지만, 실제 Test의 일반 대화는 합성 데이터와 스타일/도메인이 다름. 이는 합성 데이터의 양이나 길이가 아닌 질적 다양성의 문제.

시도한 것과 효과

시도변경효과
합성 데이터 길이 개선139자 → 206자변화 없음 (34 → 35)
합성 데이터 양 확대694건 → 1,200건미미한 개선 (34 → 38)
max_length 확대128 → 256CPU 메모리 한계로 실패

추정 원인

  1. 도메인 갭: Claude가 생성한 대화 패턴과 TUNiB이 수집한 실제 일반 대화의 어휘/문체 차이
  2. 모호한 경계: Test의 일반 대화 중 위협 키워드가 포함된 경우 (예: 돈 관련 일상, 직장 불만 등) 위협 클래스로 분류
  3. 분포 불일치: 합성 데이터가 특정 토픽에 편중되어 Test의 다양한 일상 대화를 커버하지 못함

8. 향후 개선 방향

A
합성 데이터 품질 개선

Test 데이터의 일반 대화 패턴을 역분석하여 합성 데이터 스타일 조정. 더 다양한 프롬프트 전략 적용.

B
모델 다양화

KcBERT, KLUE-RoBERTa 등 다른 모델로 비교 실험. 앙상블로 최종 성능 향상.

C
예측 임계값 조정

일반 대화 예측 확률 threshold를 낮춰서 recall 향상. Precision-Recall trade-off 분석.

D
데이터 증강

위협 클래스에 Back-translation, 문장 셔플링 적용. newline 제거 증강으로 포맷 격차 추가 해소.

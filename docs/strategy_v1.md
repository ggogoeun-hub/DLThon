## 1. 데이터 전처리 전략

### 1-1. 중복 제거 (Deduplication) — ✅ 구현 완료

> 근거: `[5-2]` 완전 동일 중복 **104건**, 준-중복(유사도≥0.95) **117건** 발견

| 작업 | 방법 | 기대 효과 |
|---|---|---|
| 완전 중복 제거 | `df.drop_duplicates(subset=['conversation'])` | 과적합 방지, 학습 효율 향상 |
| 준-중복 제거 | TF-IDF 코사인 유사도 ≥0.95인 쌍 중 하나 제거 | 모델이 '외운' 패턴에 의존하는 것 방지 |

**중복 제거 후 예상 데이터 크기**: 약 3,800건 내외

**⚠️ v1 추가**: 현재 `03_training.ipynb`에서 완전 중복만 제거 중. **준-중복 제거는 미구현** — Ablation Exp-1에서 효과 비교 필요

### 1-2. 텍스트 전처리 — ✅ 구현 완료 (방식 변경)

> 원본 전략: `\n` → `[SEP]` 토큰 변환
> **v1 변경**: `\n` → 공백 변환으로 변경

| 항목 | 원본 전략 | v1 전략 | 변경 사유 |
|---|---|---|---|
| 턴 구분자 | `[SEP]` 토큰 | 공백(space) | `[SEP]`는 BERT/ELECTRA의 세그먼트 구분 용도로 사전학습됨. 대화 턴 구분과 의미 충돌 우려 |
| 다중 공백 | 미언급 | `re.sub(r'\s+', ' ', text).strip()` | 정규화 |

**⚠️ v1 추가 실험 제안**: custom special token(`[TURN]`) 등록 후 비교 실험

```python
# 실험 코드 예시
tokenizer.add_special_tokens({'additional_special_tokens': ['[TURN]']})
model.resize_token_embeddings(len(tokenizer))
text = text.replace('\n', ' [TURN] ')
```

### 1-3. 특수문자/이모티콘 처리

> 근거: `[5-1]` 모든 클래스에서 `emoticon_len = 0.0`, `ellipsis_count = 0.0`

| 특수문자 | 처리 방침 | 근거 |
|---|---|---|
| `ㅋㅋ`, `ㅠㅠ` | **학습 데이터에 없으므로 합성 시에도 사용 금지** | emoticon_len = 0.0 |
| `...` (말줄임) | **합성 시 사용 금지** | ellipsis_count = 0.0 |
| `!` (느낌표) | **유지** — 클래스 구분 힌트 | 협박(1.85) > 갈취(1.02) |
| `?` (물음표) | **유지** | 전 클래스 4.08~4.87 고르게 분포 |

**⚠️ v1 주의**: 현재 합성 데이터(`generate_hard_negatives.py` 등)에 `ㅋㅋ`, `ㅎㅎ` 등이 포함되어 있음. 원본 train 데이터의 이모티콘 분포와 일치하는지 검증 필요. 만약 원본에 없는 패턴이면 모델이 "이모티콘 있음 = 일반 대화" 숏컷을 학습할 위험.

## 2. 데이터 증강 전략 (위협 4클래스)

> **규칙**: 새로운 데이터 생성 불가 → 기존 데이터의 **변형(Augmentation)만** 허용

### 2-1. 🚨 검증셋 사전 격리 — 증강 전 필수 *(v1 신규, v2참고 반영)*

> 증강된 데이터가 검증셋에 유사한 형태로 섞이면 F1이 부풀려지고, 리더보드 점수와 괴리 발생

1. **사전 분리**: 중복 제거된 원본(~950건/클래스)에서 **10~20%를 검증셋으로 먼저 격리**
2. **훈련셋만 증강**: 격리 후 남은 80~90%만 증강 대상
3. **검증셋은 원본 그대로**: 증강/변형 절대 불가

```python
# 증강 전 검증셋 격리
from sklearn.model_selection import train_test_split

train_base, val_base = train_test_split(
    train_dedup, test_size=0.15, random_state=42,
    stratify=train_dedup['class']
)
# val_base는 절대 증강하지 않음
# train_base만 아래 증강 기법 적용
```

### 2-2. 증강 기법 선정

| 기법 | 방법 | 적용 근거 | 비고 |
|---|---|---|---|
| **동의어 교체** | 핵심 키워드 유지, 주변 단어를 동의어로 교체 | TTR 0.90 → 소폭 교체로 새 패턴 생성 가능 | ✅ 유지 |
| ~~문장 순서 셔플~~ | ~~인접 2개 발화 swap~~ | ~~발화 횟수 median=10~~ | ❌ **삭제** |
| **패러프레이즈** *(v1 신규)* | 동일 의미를 다른 표현으로 변환 | 대화 흐름(에스칼레이션 패턴) 보존하면서 표현 다양화 | ✅ 셔플 대체 |
| **역번역** | 한→영→한 번역 | 자연스러운 표현 변형 | ⚠️ 품질 필터 추가 |
| **랜덤 삽입/삭제** | 추임새 삽입 또는 무의미 단어 삭제 | 길이 분포(41~874)가 넓어 다양한 길이 시도 가능 | ✅ 유지 |
| **복합 증강** *(v2참고 반영)* | 위 기법들을 순차/혼합 적용 | 단일 기법 한계 보완, 3,000건 목표 시 필수 | ✅ 추가 |

**🔴 v1 핵심 변경 — 문장 셔플 삭제 사유**:
- `[6-2]` 분석에서 **"후반부로 갈수록 위협 단어 밀도 급증"**을 핵심 분류 신호로 발견
  - 협박: 0.262 → 0.325 → **0.638** (후반 급증)
  - 갈취: 0.486 → 0.525 → **0.746** (후반 집중)
- 문장 셔플은 이 에스칼레이션 패턴을 파괴 → **자체 EDA 결론과 모순**
- 대체: **패러프레이즈 증강** (흐름 유지 + 표현 다양화)

원본 전략에서 역번역의 구체적 실행 방법이 부재했으므로 보완:

### 3-2. 합성 데이터의 물리적 스펙

| 파라미터 | 설정값 | 근거 |
|---|---|---|
| **생성량** | **3,000건 확정** | 별도 증강 없이 합성으로만 구성. 위협 클래스 3,000건과 균형 |

| **발화 횟수** | **10턴** (±2) | 4클래스 모두 median = 10 |
| **발화 대칭성** *(v2참고)* | **1:1 길이 대칭 (Tiki-taka)** | 위협의 비대칭성(가해자 긴 발화 / 피해자 짧은 발화) 방지. 오차 20% 이내 |

| **이모티콘/초성** | **사용 절대 금지** | `ㅋㅋ`, `ㅎㅎ` 포함 시 오분류 숏컷 발생. emoticon_len = 0.0 |
| **문장 부호** | 마침표(.) 생략 위주 | 메신저 스타일 유지 |

### 3-3. 합성 대화의 말투 설계

> 근거: `[4-2]` 존댓말/반말/비속어 통계 + 수평적 관계 분석 *(v2참고 반영)*

**일반 대화 말투 설정**:

| 파라미터 | 설정값 | 근거 |
|---|---|---|
| 존댓말 비중 | 2.0~3.0 | 위협 클래스 평균 1.39~3.22의 중간값 |
| 반말 비중 | 6.5~8.0 | 위협 클래스 평균 6.62~7.94 범위 |
| 비속어 | **0 (절대 금지)** | 일반 대화에 욕설 → 위협으로 오분류 |
| **수평적 리액션** *(v2참고)* | "맞아", "대박", "인정" 등 | 능동적 맞장구 포함. 위협의 일방적 통보와 대비 |
| **거절의 방식** *(v2참고)* | 주체적/완곡한 거절 | "난 별론데", "다음에 하자" 등. 굴복 표현 아닌 대등한 의사 표현 |
| **화법 대칭** *(v2참고)* | 상호 존대/반말 일치 | 한쪽만 존대하는 **권력 비대칭 제거** — 위협 클래스와의 결정적 차이 |

### 3-4. 합성 대화의 도메인 설계

> 근거: `[6-1]` 도메인 키워드 쏠림 분석 + 기타 괴롭힘 Topic 대응 *(v2참고 반영)*

| 도메인 | 생성 비율 | 목적 |
|---|---|---|
| **직장 일상** | 30% | `직장 = 괴롭힘` 편향 제거 |
| **학교 일상** | 20% | `학교 = 괴롭힘` 편향 제거 |
| **일상/서비스** *(v2참고)* | 20% | 정상적인 주문/문의 대화. 기타 괴롭힘 Topic 1 대응 |
| **친밀 관계** *(v2참고 신규)* | 15% | 건강한 데이트/애정표현. 기타 괴롭힘 Topic 2 대응 |
| **자유 주제** | 15% | 취미, 덕질 등 정보 밀도 높은 수다 |

### 3-5. Hard Negative 합성 — ✅ 구현 완료

> 근거: `[3-1]` 코사인 유사도 — 협박 ↔ 기타 괴롭힘 = **0.8689** (최고 혼동)

**사용 가능**: `돈이`, `제발`, `지금 당장`, `죄송합니다`, `무슨`, `진짜` → 무해한 문맥 배치
**사용 금지**: `죽여`, `죽어`, `맞을래`, `시키는 대로`, `내놔`, `닥쳐` → 절대 불포함

### 3-6. 대화 흐름(Flow) 설계

> 근거: `[6-2]` 위협 클래스는 후반부로 갈수록 위협 단어 밀도 급증

**핵심 차별점** *(v2참고 반영)*: 위협 클래스의 후반부 감정 폭발과 대비되는 **일정한 감정선(Flat Sentiment)** 유지

- **전 구간 위협/방어 밀도 = 0**: 초반부터 후반까지 일정한 텐션
- **역할 비대칭 제거**: 가해자/피해자 구조가 아닌 **대등한 상호작용**
- **감정 에스칼레이션 없음**: 위협(0.26→0.64 급증)과 대비되는 평탄한 흐름

### 3-7. 합성 데이터 자동 품질 검증 *(v1 신규)* ⚠️ 미구현

현재 합성 데이터의 품질을 자동 검증하는 파이프라인이 없음. 아래 validator 구현 권장:

```python
def validate_synthetic(df):
    """합성 일반 대화 품질 검증"""
    issues = []
    
    BANNED_WORDS = ['죽여', '죽어', '맞을래', '시키는 대로', '내놔', '닥쳐', '패버릴']
    
    for idx, row in df.iterrows():
        text = row['conversation']
        
        # 1. 길이 검증 (150~400자)
        if len(text) < 150 or len(text) > 400:
            issues.append((idx, f'길이 이상: {len(text)}자'))
        
        # 2. 턴 수 검증 (6~14턴)
        turns = text.strip().split('\n')
        if len(turns) < 6 or len(turns) > 14:
            issues.append((idx, f'턴 수 이상: {len(turns)}턴'))
        
        # 3. 금지 단어 검증
        for word in BANNED_WORDS:
            if word in text:
                issues.append((idx, f'금지 단어 포함: {word}'))
        
        # 4. 비속어 검증 (간이)
        PROFANITY = ['씨발', '병신', '개새끼', '미친놈', '꺼져']
        for word in PROFANITY:
            if word in text:
                issues.append((idx, f'비속어 포함: {word}'))
    
    print(f'검증 완료: {len(df)}건 중 이슈 {len(issues)}건')
    return issues
```

**⚠️ 추가 점검 필요**: `generate_hard_negatives.py`에 `ㅋㅋ`, `ㅎㅎ` 같은 이모티콘이 포함되어 있는데, 원본 train 데이터의 `emoticon_len = 0.0`과 불일치. 모델이 "이모티콘 = 일반 대화" 숏컷을 학습할 수 있으므로, 이모티콘 포함/제거 버전을 ablation으로 비교할 것.

## 4. 모델링 전략 *(v1 신규 섹션)*

원본 strategy.md는 데이터 전략만 다뤘으나, 모델링 결정도 EDA에 기반해야 하므로 추가.

### 4-1. 모델 선택 — ✅ 변경 완료

| 항목 | 원본 | v1 (현재) | 변경 사유 |
|---|---|---|---|
| 모델 | KLUE-RoBERTa-base | **KcELECTRA-base** | ELECTRA의 RTD 사전학습이 짧은 한국어 대화에서 더 효과적. 파라미터 수 유사하면서 추론 속도 빠름 |

**Ablation 비교 대상**: KLUE-RoBERTa-base, KcELECTRA-base, KcBERT-base

### 4-2. 하이퍼파라미터

| 파라미터 | 원본 | v1 (현재) | 변경 사유 |
|---|---|---|---|
| Epochs | 5 (고정) | **10 + EarlyStopping(patience=3)** | 5에폭은 underfitting 가능. 조기종료로 과적합 방지 |
| Batch Size | 32 | 32 (train) / 64 (eval) | 유지 |
| Learning Rate | 2e-5 | 2e-5 | 유지. Ablation에서 1e-5, 3e-5 비교 |
| MAX_LEN | 256 | 256 | ⚠️ 512 실험 추가 필요 (아래 참고) |
| Warmup | 10% | 10% | 유지 |
| Val Split | 80:20 | **85:15** | 데이터가 적으므로 학습 데이터 확보 우선 |

### 4-3. MAX_LEN 실험 *(v1 신규)*

> 근거: 현재 256 토큰에서 커버율 ~75%. 협박 클래스는 최대 874자로 잘림 위험 큼.

| 실험 | MAX_LEN | 예상 커버율 | 메모리 | 비고 |
|---|---|---|---|---|
| A (현재) | 256 | ~75% | 기본 | 빠른 학습, 긴 대화 잘림 |
| B | 384 | ~90% | 1.5× | 절충안 |
| C | 512 | ~97% | 2× | MPS에서 batch_size 축소 필요 → gradient accumulation |

```python
# gradient accumulation으로 메모리 보완
training_args = TrainingArguments(
    per_device_train_batch_size=16,       # 32 → 16
    gradient_accumulation_steps=2,         # effective batch = 32
    max_length=512,
)
```

### 4-4. 풀링 전략 실험 *(v1 신규)*

> 근거: `[6-2]` 대화 후반부에 위협 단어가 집중됨 → [CLS] 토큰이 이 패턴을 충분히 포착하는지 의문

현재 `AutoModelForSequenceClassification`은 내부적으로 [CLS] 풀링 사용. 커스텀 모델로 비교 실험 가능:

| 풀링 | 방법 | 기대 효과 |
|---|---|---|
| [CLS] (현재) | 첫 토큰의 hidden state | 표준. 빠른 baseline |
| **Mean Pooling** | 모든 토큰의 hidden state 평균 | 대화 전체 정보를 고르게 반영 |
| **Attention Pooling** | 학습 가능한 가중 평균 | 중요한 턴(후반부 위협)에 가중치 부여 가능 |

### 4-5. 손실 함수 *(v1 신규)*

| 손실 함수 | 적용 상황 | 비고 |
|---|---|---|
| CrossEntropy (현재) | 클래스 균형일 때 | 기본 |
| **Focal Loss** | 혼동 클래스 쌍 완화 | 협박↔기타괴롭힘(유사도 0.8689) 구분 개선 기대 |
| **Label Smoothing** | 과적합 방지 | `label_smoothing_factor=0.1` (TrainingArguments에 옵션 존재) |

## 6. Cross-Validation 전략 *(v1 신규)*

> 현재: 단일 85:15 split → ~4,000건 데이터에서 분산이 큼

| 전략 | 설명 | 비고 |

for fold, (train_idx, val_idx) in enumerate(skf.split(X, y)):
    # ... train & evaluate ...
    fold_scores.append(f1_macro)

1. 핵심 요약

Stage 3에서 비교한 3가지 풀링 구조 중 2개가 설계 의도대로 작동하지 않았음을 코드 레벨에서 확인.

모델설계 의도실제 작동판정
Multi-head 대등성/마무리/위협을 각각 다른 관점에서 분석 같은 벡터에서 스칼라 3개 출력 불일치
Speaker-aware 화자 A/B를 구분하여 권력 비대칭 감지 토큰 포지션 짝수/홀수 분리 불일치
Ending Pooling 대화 마무리 톤 중점 분석 마지막 64토큰만 풀링 일치

2. Multi-head 분류기 — 무엇이 잘못되었나

설계 의도

"대등성", "마무리 톤", "위협 의도"를 별도 헤드로 각각 예측한 후, 이 신호들을 결합하여 최종 분류

실제 코드

def forward(self, input_ids, attention_mask):
    out = self.backbone(input_ids, attention_mask)
    pooled = self.mean_pooling(out.last_hidden_state, attention_mask)  # 768d

    eq     = self.equality_head(pooled)   # 같은 pooled → 스칼라 1개
    end    = self.ending_head(pooled)     # 같은 pooled → 스칼라 1개
    threat = self.threat_head(pooled)     # 같은 pooled → 스칼라 1개

    return self.final_classifier(concat([pooled, eq, end, threat]))
    # 768 + 1 + 1 + 1 = 771차원 → 분류

문제 분석

문제 1: 같은 입력

3개 헤드가 동일한 mean pooled 벡터를 입력으로 받음. "대등성", "마무리", "위협"을 각각 다른 관점에서 보는 것이 아니라, 같은 정보에서 서로 다른 스칼라만 뽑아냄.

문제 2: Mean Pool의 한계

Mean Pooling은 전체 토큰을 하나로 평균. "마무리 톤"을 보려면 마지막 부분만, "대등성"을 보려면 전반/후반을 비교해야 하는데, 전부 같은 평균 벡터를 봄.

결과: 3개 헤드는 사실상 nn.Linear(768, 1) 3개 — 771차원 분류기에 3차원을 추가한 것과 같음. "서로 다른 관점"이 아니라 단순한 차원 확장 효과만 있었음.

보완: 각 헤드가 다른 영역을 봐야 함

// 올바른 설계
equality_head:
  first_half  = mean_pool(hidden[1 : L//2])        // 전반부
  second_half = mean_pool(hidden[L//2 : L-1])      // 후반부
  → 두 벡터 비교 (concat + diff)                     // 흐름 변화 감지

ending_head:
  ending = mean_pool(hidden[L-64 : L])              // 마지막 64토큰만
  → 마무리 톤 분석                                    // 이미 Ending Pooling에 있음

threat_head:
  전체 mean_pool → 위협 키워드 존재 여부 스칼라        // 현재와 동일 (OK)
핵심: 각 헤드가 시퀀스의 다른 부분을 입력으로 받아야 의미 있는 멀티 관점 분석이 가능. 현재 구조는 같은 입력에 다른 가중치만 적용한 것.

3. Speaker-aware 구조 — 무엇이 잘못되었나

설계 의도

화자 A와 화자 B의 발화를 분리 풀링하여 권력 비대칭(한쪽이 일방적으로 위협/굴복하는지)을 감지

실제 코드

pos = torch.arange(S)                              // 0, 1, 2, 3, 4, 5, ...
om = ((pos % 2 == 0) & (attention_mask == 1))       // 짝수 포지션 → "화자A"
em = ((pos % 2 == 1) & (attention_mask == 1))       // 홀수 포지션 → "화자B"

a = mean_pool(hidden * om)                          // 짝수 토큰 평균
b = mean_pool(hidden * em)                          // 홀수 토큰 평균
pooled = concat([a, b])                             // 768*2 = 1536차원

문제 분석

실제 토큰 배치를 보면:

"야   빨리  타라  늦으면  다   죽어  내가  운전할"
 pos0  pos1  pos2  pos3  pos4  pos5  pos6  pos7
  A     B     A     B     A     B     A     B

토큰 하나하나를 번갈아 A/B로 배정. 이건 화자 교대가 아님:

  • "야 빨리 타라 늦으면 다 죽어"가 한 사람의 말이면, 이 6개 토큰이 전부 같은 화자인데 A/B로 분산됨
  • 턴 경계 정보가 없으므로 어디서 화자가 바뀌는지 알 수 없음
  • 결과적으로 "짝수 토큰 평균 vs 홀수 토큰 평균" = 의미 없는 분할
결과: "화자 구분"이 아니라 "토큰 서브샘플링" 2가지 버전의 평균을 구한 것. 권력 비대칭 감지와 무관. 그런데도 성능이 나온 건 2가지 풀링을 concat한 것이 일종의 앙상블 효과를 준 것으로 추정.

근본적 한계: 턴 경계 데이터가 없음

데이터턴 구분진짜 Speaker-aware 가능 여부
Train 위협 (원본)\n 100%가능 — \n 기준으로 화자 분리
Train 일반 (합성)없음 0%불가능
Test\n 5%, 구두점 81%거의 불가능

Train에서 화자를 구분해 학습하더라도 Test에서는 턴 경계가 없어 추론 시 활용 불가. Train-Test 형식 불일치가 근본 문제.

5. 보완 방향 — 청크 흐름 구조

Test 데이터의 현실

Test 어절 경계 수단비율
구두점 뒤 공백 (마침표/물음표)13.4%
순수 공백만 (턴 경계 단서 없음)86.6%

Test의 86.6%가 턴 구분 없는 연속 텍스트 → 턴 구분 없이도 작동하는 구조가 필요

청크 흐름 구조 (현재 실험 진행 중)

RoBERTa backbone
    ↓
토큰 시퀀스 (1 ~ L)
    ↓
┌──────────────────┐  ┌──────────────────┐
│  전반부 풀링      │  │  후반부 풀링      │
│  (1 ~ L/2)       │  │  (L/2 ~ L-1)     │
│  → first (768d)  │  │  → second (768d) │
└──────────────────┘  └──────────────────┘
          ↓                    ↓
      diff = second - first (768d)
          ↓
  concat [first, second, diff] → 2304d
          ↓
      512 → 128 → 5클래스

왜 청크 흐름이 적합한가

같은 키워드, 다른 흐름:

일반: "새끼야 살이나빼"(전반)  → "팩트라 할말없네"(후반)     = 수긍 ✓
위협: "새끼야 살이나빼"(전반)  → "죄송합니다 그만해주세요"(후반) = 굴복 ✗

전반부 벡터와 후반부 벡터의 차이(diff)가 수긍/협력 vs 굴복/강압의 방향을 나타냄.

기존 대비 개선점

기존 Multi-head청크 흐름
입력같은 mean pool 벡터 1개전반/후반 + diff 3개
관점차원 추가 효과만시간적 흐름 비교
턴 구분 필요불필요 (어차피 안 씀)불필요 (토큰 절반 분할)
Train-Test 일관성일관 (둘 다 턴 없음)일관 (둘 다 절반 분할)
청크 흐름은 턴 구분이 없어도 작동하면서, 대화의 시간적 변화(전반→후반)를 명시적으로 모델링. 현재 구두점 제거 + 청크 흐름 실험이 진행 중이며, 기존 구두점 제거 + Mean Pool(75건) 대비 개선 여부를 확인 예정.

6. 교훈

1. 구조의 이름이 아니라 코드를 봐야 한다
"Speaker-aware"라는 이름이 화자 분리를 의미하지 않음. 실제 코드가 pos % 2로 토큰을 나누고 있으면, 그건 화자 분리가 아니라 서브샘플링.

2. 같은 입력에서 다른 출력은 "다른 관점"이 아니다
진정한 멀티 관점은 다른 영역을 보는 것. 같은 mean pool에 다른 Linear를 적용하는 건 관점이 아니라 차원 확장.

3. Train-Test 형식 일관성이 전제 조건
Train에서만 가능한 피처(턴 구분, 화자 구분)를 학습해도 Test에서 활용 불가. 설계 시 Test 조건을 먼저 확인해야 함.

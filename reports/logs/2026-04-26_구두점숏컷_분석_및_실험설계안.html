1. 문제 정의 — 왜 일반대화가 88건에서 멈추는가

현재 최고 모델(M3-D Multi-head, 6ep)이 Test 일반대화 100건 중 88건만 정분류. 나머지 12건은 위협으로 오분류됨.

마침표 개수별 일반대화 예측 비율

마침표 개수Test 샘플 수일반대화 예측비율
0개152건77건50.7%
1~2개65건0건0.0%
3~5개101건6건5.9%

88건 중 77건(87.5%)이 마침표 0개인 샘플. 마침표가 1개만 있어도 일반대화 예측이 0%로 떨어짐. 모델이 의미가 아닌 구두점 개수를 세서 분류하고 있다는 결정적 증거.

3. 토크나이저 레벨 분석 — 구두점이 모델에 어떻게 보이는가

RoBERTa 임베딩 구조

입력 텍스트 → Token Embedding (32,000 × 768)  "의미"
            + Position Embedding (514 × 768)  "위치"
            → LayerNorm → 12층 Self-Attention
            → Mean Pooling → 768차원 벡터 → 분류 헤드

구두점 토큰의 정체

입력토큰화구두점 토큰
위협 (마침표 有) ...왜그러세요 | . | 여기...죽을래 | ? | 잘못했어요 51토큰 (구두점 2개)
일반 (마침표 無) 야 | 빨리 | 타라 | 늦으면 | 다 | 죽어 | 내가 | 운전할 48토큰 (구두점 0개)
.과 ?는 각각 독립 토큰 1개로 처리됨. 의미적으로는 "." vs "?" 코사인 유사도 0.14로 거의 무관한 토큰.

토크나이저는 의미 단위를 잘 분리하고 있음:
"떼먹는다고" → 떼 | ##먹 | ##는 | ##다고 — 의미 충분
"죽여버린다" → 죽여 | ##버린 | ##다 — 의미 충분

구두점은 의미를 추가하는 게 아니라, 모델이 의미 대신 세는 숏컷 피처를 제공하여 학습을 방해.

토큰 길이 분포 — 또 다른 숏컷

4. 청크 흐름 분석 — 의미 기반 분류의 본질

개별 키워드가 아닌 청크 간 관계(흐름의 변화)가 일반 vs 위협을 결정:

"새끼"가 들어가도 청크 흐름이 다름

일반 (쌍방 디스 → 수긍)

"이 새끼가 맨날"  →  "처먹고 운동도 안하니까"  →  "팩트라 할말이 없네"
     [비난 청크]          [근거 청크]                  [수긍 청크] ✓

괴롭힘 (일방 비하 → 배제)

"촌스러워 걔 원래"  →  "저래 무시해 그만해"  →  "싫으면 학교를 오지마"
     [비하 청크]            [동조 청크]               [배제 청크] ✗

"죽어"가 들어가도 청크 맥락이 다름

일반 (게임)

"늦으면 다 죽어"  →  "내가 운전할 테니까"  →  "너는 총 쏘는 거 집중해"
     [게임 상황]          [협력 제안]               [역할 분담] ✓

위협 (실제)

"진짜 죽여버린다"  →  "시키는 대로 안 하면"  →  "가만 안 둬"
     [직접 위협]           [조건부 강압]             [위협 강화] ✗
핵심은 후반부가 어떻게 전개되느냐 — 수긍/협력이면 일반, 굴복/강압이면 위협. 현재 Mean Pooling은 이 흐름을 섞어버림.

5. Stage 3 모델 구조 재평가 — 설계 의도 vs 실제 구현

모델설계 의도실제 구현평가
Speaker-aware 화자 A/B 분리 → 권력 비대칭 감지 토큰 포지션 짝수/홀수 분리 의도와 불일치
토큰 단위 교대 ≠ 화자 교대. 턴 경계 없이 화자 구분 불가능
Multi-head 대등성/마무리/위협을 각각 분석 같은 mean pool 벡터에서 스칼라 3개 출력 의도와 불일치
3개 헤드가 같은 입력 → 서로 다른 관점 아님. 차원 추가 효과만
Ending Pooling 마무리 톤 중점 분석 마지막 64토큰만 풀링 의도대로 작동
유일하게 다른 영역을 보는 구조
Stage 3에서 비교한 것은 "서로 다른 풀링 전략"이 아니라 Mean Pool + 약간의 변형 3가지. 88건 vs 85건 vs 84건의 차이는 구조적 우위가 아닌 학습 노이즈 수준일 수 있음.

EXP-1: 구두점 제거 현재 진행 중

가설구두점이 숏컷 피처 → 제거 시 모델이 의미 기반 분류로 전환
방법전처리에서 . ? ! 제거 후 동일 조건 학습
기대효과마침표 1+개 샘플에서 일반대화 정분류율 상승 (현재 3.2% → 목표 15%+)
위험위협 데이터 턴 경계 소실 → 위협 클래스 성능 하락 가능
예상 영향가장 큼 — 병목의 근본 원인(숏컷) 제거

EXP-2: 청크 흐름 구조 (전반/후반 비교)

가설대화의 전반→후반 변화(수긍 vs 굴복)가 분류의 핵심 신호
방법토큰 시퀀스를 절반으로 나눠 각각 풀링 → 두 벡터의 관계로 분류
구현
first_half = mean_pool(hidden[:, :L//2, :])   # 전반부
second_half = mean_pool(hidden[:, L//2:, :])  # 후반부
flow = concat([first_half, second_half, first_half - second_half])
→ classifier(flow)
기대효과중간 — 경계 샘플(장난 vs 실제 괴롭힘)에서 개선
전제조건EXP-1(구두점 제거) 위에 적용해야 효과 분리 가능

EXP-3: 턴 구분 특수 토큰 삽입

가설턴 경계 정보가 화자 간 역학 파악에 도움
방법구두점 대신 [SEP]를 턴 경계에 삽입
데이터위협: 원본 \n → [SEP] 변환 가능 (100% 보존)
Test: 마침표 기반 턴 추정 (66.4%)
일반: 턴 경계 없음 → 새 숏컷 위험
핵심 과제일반대화에 턴 경계가 없어서 비대칭 문제. 합성 데이터 재생성 또는 휴리스틱 턴 추정 필요
기대효과중~낮음 — 데이터 재구축 부담 대비 효과 불확실
왜 이 순서인가: EXP-1이 병목(숏컷)을 제거하는 것이고, EXP-2/3은 모델 능력을 강화하는 것. 병목이 있는 상태에서 능력을 올려봐야 효과가 제한적이므로, 병목 해소 → 능력 강화 순서가 최적.

7. 여기까지의 핵심 교훈

1. 데이터 분포가 모델 능력을 압도한다
구두점 분포 불일치 하나가 모델 구조 변경(3가지), 에폭 증가(3→6), 동결 해제보다 더 큰 영향. 모델을 바꾸기 전에 데이터를 먼저 봐야 한다.

2. 설계 의도와 실제 구현을 검증해야 한다
Speaker-aware(토큰 짝홀수 ≠ 화자 구분)와 Multi-head(같은 벡터에서 3개 스칼라 ≠ 다른 관점)가 설계 의도대로 작동하지 않았음. 구조를 만들었으면 실제로 의도한 정보를 보고 있는지 검증이 필요.

3. 숏컷은 겉으로 보이지 않는다
Val F1 0.91, Test 88건이면 "잘 작동한다"고 생각하기 쉽지만, 마침표별 분류율을 보면 87.5%가 숏컷 의존. 성능 지표만으로는 모델이 정말 의미를 이해하는지 알 수 없다.
